## 二、可复用的产品需求提示词模板

下面这部分是从上面的原始沟通中提炼出来的一套更适合复用的产品需求提示词模板。后续如果要做类似项目，可以直接基于这一版继续扩展。

### 1. 项目总提示词

```text
我想基于一份历史文本材料，做一个“古代地缘形势推演”的可视化网站。

目标不是做普通古籍阅读器，而是做一个让用户直观看懂：
1. 各区域的地缘位置
2. 攻守关系
3. 关键节点
4. 战略通道
5. 风险与优势
6. 古代地名和现今地区的对应关系

网站底层基于 Next.js，优先使用成熟的开源图形库来完成地图、关系图、拓扑图和指标可视化。

在开始写代码前，先整理好技术选型、信息架构、数据结构和页面结构；确认方案后再开发。
```

### 2. 数据整理提示词

```text
请先把原始文本中的各区域序文提取出来，整理成一个 Markdown 文件。

要求：
1. 保留每个区域的独立标题和正文
2. 去掉不必要的来源信息
3. 方便后续程序解析
4. 结构清晰，适合后续转成结构化数据
```

补充说明：

```text
数据取材于《读史方舆纪要》，但页面表达不要过度围绕书本本身，而要围绕产品功能和用户理解。
```

```text
这个产品的定位是“古代地缘形势推演工具”，不是“古籍介绍页面”。

用户关心的是：
1. 这个地区为什么重要
2. 为什么能守
3. 为什么能攻
4. 哪些地方是屏障、门户、通道、腹地
5. 不同区域之间的战略关系是什么

所以页面文案、标题、导航和 README 都要围绕“地缘形势推演”来写，而不是围绕书籍作者和古籍背景来写。
```

### 17. 移动端适配提示词

```text
请重点优化移动端体验，不要只是把桌面布局缩小。

移动端设计原则：

1. 首屏高度很宝贵，避免大块目录、搜索框、侧栏列表占据主要内容之前的位置。
2. 移动端进入首页和详情页时，都应优先看到当前内容标题、视图切换按钮和核心内容。
3. 如果有多地区、多章节入口，移动端优先做成横向滚动的小标签导航，而不是纵向卡片列表。
4. 标签文案要短，例如“总览、北直、南直、山东”，不要重复“地缘形势”这类长后缀。
5. 当前页面对应的标签需要高亮，用户要能一眼知道自己在哪里。
6. 搜索框如果不是移动端核心任务，移动端隐藏；可以只在桌面侧栏保留。
7. 桌面端可以保留完整侧边栏目录；移动端和桌面端不必使用同一种导航形态。
8. 详情页不要先展示一大块其它章节入口，避免把当前页面内容挤到首屏下方。
9. 所有图表、拓扑图、地图、正文容器都要检查横向溢出，避免固定宽度撑破 390-430px 宽度。
10. 按钮、标签、卡片需要有稳定尺寸，文字不能挤压、重叠或撑开布局。
11. 移动端正文和标题字号要克制，避免桌面级大标题浪费首屏空间。
12. 修改后必须在真实移动视口下检查，不只看代码是否响应式。

核心判断：

移动端不是“桌面端窄一点”，而是要重新判断信息优先级。桌面端可以展示完整导航和辅助工具，移动端要优先服务当前阅读和当前操作。
```
