## 二、可复用的产品需求提示词模板

下面这部分是从上面的原始沟通中提炼出来的一套更适合复用的产品需求提示词模板。后续如果要做类似项目，可以直接基于这一版继续扩展。

### 1. 项目总提示词

```text
我想基于一份历史文本材料，做一个“古代地缘形势推演”的可视化网站。

目标不是做普通古籍阅读器，而是做一个让用户直观看懂：
1. 各区域的地缘位置
2. 攻守关系
3. 关键节点
4. 战略通道
5. 风险与优势
6. 古代地名和现今地区的对应关系

网站底层基于 Next.js，优先使用成熟的开源图形库来完成地图、关系图、拓扑图和指标可视化。

在开始写代码前，先整理好技术选型、信息架构、数据结构和页面结构；确认方案后再开发。
```

### 2. 数据整理提示词

```text
请先把原始文本中的各区域序文提取出来，整理成一个 Markdown 文件。

要求：
1. 保留每个区域的独立标题和正文
2. 去掉不必要的来源信息
3. 方便后续程序解析
4. 结构清晰，适合后续转成结构化数据
```

补充说明：

```text
数据取材于《读史方舆纪要》，但页面表达不要过度围绕书本本身，而要围绕产品功能和用户理解。
```

### 3. 产品定位提示词

```text
这个产品的定位是“古代地缘形势推演工具”，不是“古籍介绍页面”。

用户关心的是：
1. 这个地区为什么重要
2. 为什么能守
3. 为什么能攻
4. 哪些地方是屏障、门户、通道、腹地
5. 不同区域之间的战略关系是什么

所以页面文案、标题、导航和 README 都要围绕“地缘形势推演”来写，而不是围绕书籍作者和古籍背景来写。
```

### 4. 核心功能提示词

```text
请围绕以下几个核心视图来设计产品：

1. 地理视图
- 不是几个散点
- 要有地图感
- 最好能体现地势高低
- 用户能看出山河、通道、屏障、腹地

2. 推演视图
- 用拓扑结构展示核心、屏障、门户、通道、资源地、外部威胁
- 要能体现地缘战略关系
- 每个省都要有，不只是单个示例省份

3. 关系视图
- 展示地区、主题、节点之间的连接关系

4. 形势视图
- 展示权重、守势、机动、风险等指标

5. 原文视图
- 保留原始文本入口，方便对照
```

### 5. 交互细节提示词

```text
请补充以下交互能力：

1. 默认打开“推演”tab
2. 古代地名需要能看到对应现今地区
3. 不要把古今地名说明直接塞满画面
4. 更适合用 hover tooltip 显示现今对应地区
5. 如果某块信息已经能通过 hover 获得，就不要在侧栏重复堆一份
6. 页面布局要避免重复表达同一层信息
```

### 6. 视觉与排版提示词

```text
请采用偏克制、学术感但可用性高的视觉风格。

要求：
1. 全站统一使用霞鹜文楷
2. 避免中英文混杂的界面标签，用户可见文案尽量中文化
3. 标题、主标题、面板标题、正文、辅助标签之间要有清晰字号层级
4. 不要让多个区域都像一级标题一样抢视觉中心
5. 避免白字落在白背景附近导致看不清
6. 节点内部文字要兼顾美观与可读性，优先保持整体性，不要轻易破坏图的观感
7. 去掉没有价值的装饰或冗余元素，比如无意义序号、重复信息块、不必要的小地图
```

### 7. 文案优化提示词

```text
请把产品中的文案统一处理为：

1. 页面标题更产品化
- 例如“古代地缘形势推演”
- 不要沿用“某某可视化”这种过于笼统的叫法

2. 区域标题改成白话表达
- 例如“北直方舆纪要序”改成“北直地缘形势”

3. 核心判断改成白话文
- 用户不需要自己翻译文言判断句
- 要直接读懂结论

4. README 以产品能力为主
- 重点写这是一个古代地缘形势推演工具
- 提及数据取材于《读史方舆纪要》即可，不要花太多篇幅讲作者和书本背景
```

### 8. 布局优化提示词

```text
请优化页面布局，重点注意：

1. 画布区域要充分利用屏幕空间
2. 不能出现画布很小、下面却空一大片的情况
3. 如果画布变大，节点布局和元素尺寸也要同步适配
4. 右侧信息栏只保留真正必要的信息
5. 同一页面中不要让“摘要 / 核心判断 / 推演判断”重复表达相同内容
6. 如果“核心判断”更重要，可以把它放到推演图顶部的关键位置
```

### 9. 开源项目提示词

```text
这个项目将来要开源，请补齐开源项目基础设施：

1. 生成中文 README
2. README 重点写产品定位、功能、技术栈、运行方式、数据结构、后续计划
3. 添加合适的开源许可证
4. 页面中可以适当加入 GitHub 仓库入口，但不要喧宾夺主
5. 文案整体保持中文一致性
```

### 10. 工程执行提示词

```text
在执行过程中请遵循以下方式：

1. 先整理技术选型，再开始开发
2. 做完一个阶段就给出结果
3. 遇到文案、布局、交互问题时，优先按产品逻辑收敛，而不是机械堆功能
4. 每次完成后可以直接提交、推送
5. 需要时部署到 Vercel，并保证页面 title、description、README、许可证都和产品定位一致
```
